WAV → SRT：语音识别到字幕的组织逻辑

本文档描述 sense-voice 仓库如何把 VR/长音频 WAV 转成可烧录的 SRT。 核心思想参考 KrillinAI：先文本、后时间—— 不在 VAD 粗粒度时间上直接烧字幕，而是用字符级时间戳对齐、润色后再回对齐。

1. 总览：两阶段流水线

Mac 上开发代码 → push GitHub → mag（GPU）跑测试 → rsync 拉回 SRT 与指标报告。

ASR / 音频 时间轴算法 本地 LLM（Ollama） 产物

2. 与 KrillinAI 的对应关系

KrillinAI（Whisper）

  1. Whisper 词级 Word{Text, Start, End}
  2. 标点 + LLM 拆句（不带时间）
  3. TimestampGenerator 字符串对齐
  4. MaxWordOneLine 行内再切
  5. 写出 SRT

sense-voice（SenseVoice + FunASR）

  1. output_timestamp=True → 2751 字符级时间戳
  2. tag 组对齐 + 标点拆句（subtitle.py）
  3. timestamps.py 字符流对齐（移植 KrillinAI）
  4. srt.py：≤12 字 / ≤6s 显示切分
  5. .asr.srt + .zh.srt

3. 阶段详解

阶段 A — 音频 → 结构化 ASR（transcribe.py）

每个 tag 组得到一条 segment：{start, end, text, lang, emotion, timing_confidence}。

阶段 B — 拆句与细化（subtitle.py）

对超长 segment（>8s 或加权字数 >40）按标点 。！？!?. 和逗号再切，并用 match_sentence_timestamp 重新对齐。

日文 VR 口语常无句号，因此 segment 数可能仍为 122；显示级切分在下一阶段完成。

阶段 C — LLM 润色（benchmark_voice_llm.py）

紧凑时间线送入 Ollama（如 nsfw-local:27b），profile：

  • vr — 场景整理 + 中译
  • subtitle — 只做断句/去口癖，不改语义，严格 1:1 条目
输入: [00:02.4-00:13.3] 惜しくなっちゃった 私たちのことも…
输出: - [00:02.4-00:13.3] 真可惜，也让我们的身体感到舒适吧…

阶段 D — 润色后重对齐（P1）

阶段 E — 写出 SRT（srt.py）

1
00:00:02,430 --> 00:00:13,350
惜しくなっちゃった 私たちのことも気持ちよくして ほら、入れてあげて。

2
00:00:17,770 --> 00:00:23,350
あった入っちゃた。

每次测试产出两个文件：

  • .asr.srt — 原文时间轴，用于检查切分是否准确
  • .zh.srt — 中文字幕，用于烧录

4. 字符对齐算法（timestamps.py）

若精确子串匹配失败，回退到 fuzzy_match_sentence（按字符集合在 words 中找连续区域）。

7. 如何复现

# mag 上
cd ~/work/voice_understanding && git pull
pixi run vr-subtitle-test -- \
  --clips test_voice_clips --name-filter kavr \
  --report reports/subtitle/metrics/kavr_p1_subtitle_metrics.md \
  --run-label my_run \
  --asr-backend official --asr-device cuda:0 \
  --polish-model nsfw-local:27b --language ja --profile subtitle

# Mac 上拉回
pixi run sync-reports-from-mag
